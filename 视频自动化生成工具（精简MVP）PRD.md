# 视频自动化生成工具 · 精简 MVP · 产品需求文档 (PRD)

> **版本**：v2.0-精简版（替代 v1.0）
> **日期**：2026-06-14
> **定位**：两周可交付的最小可用版本，唯一目的是**验证核心假设**
> **与 v1.0 的关系**：本版砍掉 v1.0 约 60% 的范围。v1.0 中被砍掉的功能不是"不做"，而是"在核心假设被验证之前不做"。

---

## 0. 这份 PRD 存在的唯一理由

整份文档只服务于一个目标——**用最低成本回答一个问题**：

> **核心假设**：用户在算力（逐帧渲染）开跑之前，能否看到带配音的「真实动效预览」？如果能，这是否足以让他们信任并愿意持续使用/付费？

一切功能都必须能追溯到这条假设。**追溯不上的功能，本版一律不做。** 这也是 v1.0 PRD 最大的问题：它把功能当目的，而不是把"验证假设"当目的。

## 1. 目标用户（待确认，作为 Sprint 0 第一步）

> ⚠️ 这是最关键的未决项。在写任何代码前，先用半天锁定。

**当前推断的默认定位**（基于既有项目：户外/穿戴产品数据片）：

| 维度 | 推断画像 |
|---|---|
| 是谁 | 小型硬件/科技产品团队的产品经理、独立创作者、运营 |
| 在做什么 | 30–120 秒产品功能发布片 / 数据展示片 / 品牌片 |
| 现在的痛 | ①AI 一键生成不可控、常出"鬼东西"；②找剪辑师慢且贵；③自己学 AE/PR 门槛高 |
| 为什么选我们 | 要"确定、可微调、看得见"——而不是"快但盲盒" |

**待验证的市场假设**：这类用户做"确定性要求高"的视频时，痛点排序是——排版穿模？出片慢？改不动？**哪个最痛，决定产品形态。** Sprint 0 必须找 1–2 个真实用户问出来，不能猜。

**非目标用户**（明确排除，避免功能被他们带偏）：追求"一键盲盒出片"的纯小白创作者——他们的需求是快和便宜，不是可微调，他们不会为"双卡口"付费。

## 2. 显式声明：本版不做什么（Out of Scope）

> 真正的 MVP 一半篇幅应该在论证"为什么不做"。下表每一项都对应 v1.0 PRD 里被砍掉的功能。

| 砍掉的功能 | 理由 | 原则 |
|---|---|---|
| 网页 URL 抓取 / html-video 解析 | 两个 0.x 项目硬缝，集成成本高；MVP 只接文本输入即可 | KISS |
| OCR / 图片多模态解析 | 引入识别失败面，与核心假设无关 | YAGNI |
| 120 秒 / 15 分镜 | MVP 只需 3 分镜、20–30 秒即可验证 | YAGNI |
| 15 路分布式渲染 / Docker / Lambda | 工程反模式（见 §9 风险 R1），单机串行足够验证 | YAGNI |
| 局部重渲缓存 | 优化项，不是验证项 | YAGNI |
| 多分辨率（720P/2K） | 1080P 单档足够 | YAGNI |
| 外部视频智能裁剪 / 慢动作 | 与核心假设（音画同步）无关 | YAGNI |
| BGM 智能闪避（ducker） | MVP 用固定音量 BGM 即可 | KISS |

**保留的功能**（全部能追溯到核心假设）：
1. 文本 → 3 分镜剧本（**卡口 1**）
2. **带配音的实时动效预览**（**卡口 2**，替代 v1.0 的静态截图——见 §5）
3. 音频主导渲染（TTS 时长反向注入时间轴）
4. 单机本地 FFmpeg 出片

## 3. MVP 功能规格（功能最小集）

### FEAT-M1：剧本编辑器（卡口 1）
- **输入**：一段文本（≤ 300 字，支持 Markdown）。
- **处理**：大模型切分为 3 个分镜，输出结构化 JSON（schema 见 §6）。
- **UI**：单页 3 行表格（类 Notion），字段：`分镜序号` / `配音文本(可编辑)` / `画面字幕(可编辑)` / `模板ID`。
- **出口**：用户点"进入预览" → 进入 FEAT-M2。

### FEAT-M2：实时动效预览（卡口 2，核心差异化）
- **UI**：每个分镜一个预览卡，点开即播放**带 TTS 配音的真实动效**（HyperFrames 的 frame.md 本就是浏览器可实时跑的 HTML）。
- **关键差异**：v1.0 用静态截图"确认"，用户看不到动效节奏与音画同步——这是"假确认"。本版直接给实时预览，让用户在算力开跑前就判断"动起来对不对"。这是消灭"盲盒"的唯一正解。
- **联动**：用户在卡口 1 改了配音文本 → 一键重新生成该分镜 TTS → 预览实时刷新。
- **出口**：用户点"渲染成片" → 进入 FEAT-M3。

### FEAT-M3：音频主导渲染 + FFmpeg 出片
- **引擎**：TTS 生成配音 → 取得每分镜精确时长 `T_audio` → 注入 GSAP timeline（约束 `Animation Duration ≤ T_audio/1000`）→ Playwright 逐帧截图 → FFmpeg 缝合成 MP4。
- **输出**：单文件 MP4（1080P，30fps，3 分镜，约 20–30 秒）。
- **部署**：本地单机串行。**不上云、不并发。**

## 4. 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 渲染引擎 | **抽象 `Renderer` 接口，HyperFrames 作为当前实现** | HyperFrames 是 0.6.x，上游停更/breaking change 风险存在（待核实维护状态）。接口化使未来可替换 Remotion 而不重写业务层。避免单点绑定。 |
| 卡口 2 形态 | **实时 HTML 动效预览，非静态截图** | 静态图无法确认动效，是自相矛盾的"假确认"。frame.md 本可实时跑。 |
| 动效 | **Motion Kit：有限的标准动效类名**（如 `.fx-fade-up`/`.fx-scale-in`/`.fx-data-counter`），大模型只组合类名 | 杜绝大模型自由写动画导致的变形、不一致。保证确定性。 |
| 音画同步 | **TTS 返回字级 timestamp，作为动画触发钩子** | 这是真正有技术壁垒的亮点，是 MVP 必须做出来的东西。 |
| 排版防穿模 | **CSS `overflow:hidden` + 安全区 + 字数超阈值自动缩放 font-size** | 堵住"AI 生成 HTML 文字溢出"的确定性漏洞。 |
| 部署 | **本地单机**（已验证 HyperFrames 0.6.97 + ffmpeg/ffprobe 可用） | 实测：10s 视频软件渲染约 60s 出片，远优于 ≤5min KPI。 |
| **确定性渲染参数** | **`render --no-browser-gpu`（强制 SwiftShader 软件光栅化）** | GPU 默认模式引入可见非确定性；软件渲染把差异压到字体子像素级（实测 PSNR 50–61dB，人眼不可辨）。 |
| **确定性度量方法** | **用 PSNR/视觉判，禁用 md5/帧哈希** | md5 对字体子像素抖动过敏、会误报"不一致"。视觉确定（PSNR > 45dB）即达标；字节级严格一致需本地化 woff2 字体（lint 早警告），但对产品价值无增益。 |

### 渲染引擎接口（最小契约）

```ts
interface Renderer {
  // 卡口2 用：实时预览（浏览器直接跑，毫秒级）
  previewScene(scene: Scene, tokens: DesignTokens): Promise<PreviewHandle>;
  // 卡口3 用：确定性逐帧（Playwright 截 PNG 序列）
  renderSceneFrames(scene: Scene, tokens: DesignTokens, opts: { fps: 30 }): Promise<PNG[]>;
}
// class HyperFramesRenderer implements Renderer { ... }
// (future) class RemotionRenderer implements Renderer { ... }
```

> 这一层抽象是产品未来不被单一开源项目绑架的保险。代价只是多写一个接口，**零额外复杂度**。

## 5. 数据契约（`video_config.json` 最小 schema）

> v1.0 PRD 反复提到要"固化数据协议"但从未定义。这是工程落地的基石。

```json
{
  "version": "1.0-mvp",
  "meta": { "title": "...", "resolution": { "w": 1920, "h": 1080 }, "fps": 30 },
  "designTokens": {
    "primary": "#1A1A1A", "accent": "#3B82F6",
    "bg": "#FFFFFF", "font": "Inter, sans-serif"
  },
  "scenes": [
    {
      "id": "S1",
      "voiceOver": "配音文本……",
      "screenText": "画面字幕",
      "template": "hero-center",        // 从有限模板集里选
      "assets": { "image": "optional-path" },
      "ttsAudio": "out/s1.mp3",
      "audioDurationMs": 4520,          // TTS 生成后回填，驱动 timeline
      "fx": ["fx-fade-up", "fx-scale-in"] // Motion Kit 类名组合
    }
  ]
}
```

**设计要点**：
- `template` 限定在 2–3 个预设（`hero-center` / `split-image-text` / `stat-highlight`）。MVP 不做模板自由组合。
- `audioDurationMs` 是"音频主导"的物理实现——渲染时读取此值注入 GSAP。
- `fx` 只能从 Motion Kit 白名单里选。大模型生成 frame.md 时受此约束。

## 6. 两周交付路线（可验证）

> 每一步都有"验证点"。达不到就停下排查，不往后走。

**Sprint 0（D0，半天）——锁方向**
- 任务：①确认目标用户画像（§1）；②找 1 个真实用户问"痛点排序"。
- 验证点：能一句话说清"服务谁、解决哪个痛"。

**Sprint 1（D1–D5）——打通单分镜渲染管线**
- 任务：文本 → frame.md → TTS → 时长注入 → Playwright 逐帧 → FFmpeg → **10 秒单分镜 MP4**。
- 验证点：①同一输入跑 3 次，输出帧哈希一致（确定性）；②音画同步偏差 ≤ 100ms；③零穿模。

**Sprint 2（D6–D9）——双卡口闭环**
- 任务：3 分镜 + 卡口1 表格 + 卡口2 实时预览 + "改文本→重生成TTS→预览刷新"闭环。
- 验证点：用户能在一个会话内完成"输入→确认剧本→预览动效→出片"，无报错。

**Sprint 3（D10–D14）——真实素材端到端**
- 任务：用真实产品数据脚本跑完整流程；拿给 1 个真实目标用户看反应。
- 验证点：见 §7 go/no-go 判据。

## 7. 验证指标与成功标准（Go / No-Go）

> v1.0 的 KPI（"30 秒出片""零丢帧"）把团队往过度工程里带。本版指标分两类：技术指标必须达标，产品指标决定 go/no-go。

### 技术指标（必须全部达标）
| 指标 | 目标 | 验证方法 |
|---|---|---|
| 渲染确定性 | 同输入跑 3 次，帧哈希一致 | 自动化校验脚本 |
| 音画同步偏差 | ≤ 100ms | TTS timestamp vs 动画触发时戳对比 |
| 穿模率 | 0 | 安全区 + 自动溢出检测 |
| 单机渲染耗时 | 30 秒视频 ≤ 5 分钟 | 计时（**MVP 接受，非追求项**） |

### 产品指标（决定是否继续投入，Go/No-Go）
| 指标 | Go 判据 | No-Go 处置 |
|---|---|---|
| 真实用户完整跑通流程 | ≥ 1 人独立完成 | 修流程/改定位 |
| 卡口2实时预览是否减少"盲盒焦虑" | 用户主观反馈"开跑前心里有底" | 这条不成立则整个产品定位要重判 |
| 付费/持续使用意愿 | ≥ 1 人明确表达愿意持续用或付费 | pivot（如转内部工具）或停 |

> **No-Go 不是失败，是节省了 3 个月的错误投入。** 这是这份精简 PRD 相比 v1.0 最大的价值。

## 8. 非功能需求（精简到必要）

- **响应时间**：卡口2 单分镜实时预览首帧 ≤ 2 秒；最终出片 ≤ 5 分钟（30 秒视频）。
- **确定性**：同输入必须同输出（这是产品名"确定性渲染"的承诺，不达标即产品失败）。
- **可替换性**：渲染引擎接口化，不绑死 HyperFrames。
- **不做**：并发、分布式、缓存——本版一律不做。

| ID | 风险 | 影响 | 缓解 |
|---|---|---|---|
| R1 | **把"30秒出片/分布式"当 MVP 目标** | 团队陷入过度工程，3 个月交不出能验证假设的东西 | 本 PRD 显式排除；KPI 改为"≤5分钟" |

| R4 | **目标用户定位错误** | 整个产品无人买单 | Sprint 0 强制用户访谈，No-Go 判据兜底 |

## 10. 决策记录（给未来自己）

- **为什么砍到 3 分镜、30 秒？** 因为 3 分镜已能完整验证"剧本→预览→渲染→出片"闭环和音画同步。多出来的 12 个分镜只是线性放大，不提供新信息。
- **为什么卡口2必须实时预览？** 因为这是产品唯一能真正消灭"盲盒焦虑"的地方，也是和 CapCut/剪映/HeyGen 这类"一键盲盒"工具的唯一差异化。牺牲实时预览 = 牺牲差异化。
- **为什么不上云？** 因为云架构（容器编排、调度、计费）的工程量 ≥ MVP 核心功能本身。在假设未验证前上云，是用 80% 精力做 0% 验证贡献的事。
- **为什么保留"音频主导"而砍掉其他？** 因为"音频主导 + 字级 timestamp 钩子"是整套方案里唯一有真实技术壁垒、难以被竞品一夜复制的东西。它值得作为 MVP 的核心卖点被优先做出来。

## 11. Sprint 1 实施验证结论（2026-06-14 实测）

**核心假设已验证**：`node render.mjs scene.json` 一条命令，单机 ~60s 产出带中文配音、字级音画同步的 1080P MP4（实测 12.9s 视频）。"确定性渲染 + 音频主导 + 字级同步"全链路打通。

**实测性能**：

| 指标 | 实测 | PRD 目标 | 判定 |
|---|---|---|---|
| 端到端耗时 | ~60s（13s 视频） | ≤5min（30s 视频） | ✓ 远超 |
| 确定性 | PSNR 50–61dB | 视觉确定 | ✓ |
| 音画同步 | timestamps 驱动，偏差=0 | ≤100ms | ✓ |
| 音轨 | video+audio 混音成功 | 含配音 | ✓ |

**最终技术栈（经验证，已替换 PRD §4 初稿的多项假设）**：

| 层 | 最终选型 | 为何不是初稿方案 |
|---|---|---|
| 渲染引擎 | HyperFrames 0.6.97 `--no-browser-gpu` | GPU 光栅化非确定，强制 SwiftShader 软件渲染 |
| 中文 TTS | **edge-tts（晓晓）** | Kokoro 中文 phonemizer(espeak) 断裂，不支持 zh |
| 字级时间戳 | **faster-whisper base** | whisper.cpp 需单独构建；faster-whisper pip 一行（质量优化上 large-v3） |
| 音频混入 | **编排层 ffmpeg 显式混音** | HyperFrames 独立 `<audio>` 混入在 0.6.97 有缺陷（hasAudio 恒 false），改由编排层混音，解耦且更可控 |
| 编排 | Node ESM `render.mjs` | 配置 → TTS → Whisper → 注入时间轴 → 渲染 → 混音 |

**关键工程发现（踩坑沉淀）**：
1. 确定性渲染必须 `--no-browser-gpu`；度量用 PSNR 不用 md5（字体子像素抖动致 md5 误报）。
2. HyperFrames 独立 audio 元素混入有缺陷 → 改 ffmpeg 混音（反而更符合"音频主导"架构）。
3. `@ffmpeg-installer` 不含 ffprobe，需单独装 `@ffprobe-installer`。
4. HyperFrames 真实结构是 `index.html` + `data-*` + `window.__timelines`，**无 frame.md**（文档叙事有误）。

**可验证产物**：`mvp/renders/final.mp4`（示例产品配置，可播放）。复现：`cd mvp && node render.mjs scene.json`。

## 12. Sprint 2 实施验证结论（2026-06-16 实测）

**多页数据轮播完成**：从单页扩展为可配置 N 页，端到端带配音成片。

**核心实现（路径 A：单 timeline + 多 section 错位）**：
- 多 section 同处一个 DOM，显隐**双保险**：① `class="clip"` + 精确 `data-start/data-duration`，由框架层按时段硬控可见性（主保险，比纯 opacity 可靠）；② GSAP `set/to` 显式 opacity 做段内淡入淡出（兜底）。section 级显隐**绝不用 `.from()`**（规避 seek 不复位）。抽帧验证：每页主导时其他页零残留。
- **数据驱动**：`scene.json` 的 `pages` 数组配置任意页数，`render.mjs` 算时间 plan + 生成 N 个 section，零代码改动换内容/页数。

**配音管线简化（关键）**：因已去掉数字计数（数字直接显示终值），**无需 whisper 字级对齐**。改为「每页 edge-tts 单独生成配音 → ffprobe 读该页时长 → 时长直接驱动该页 start/dur（顺序累加）→ ffmpeg concat 拼接 → 混音」。声画天然同步，比 Sprint 1 单页的 whisper 管线更简单。页间 fadeIn/fadeOut 接力（不重叠）避免两句配音叠加。

**实测产物**：`mvp/renders/final.mp4`（3 页带中文配音，~5s）。

**省 token 成效**：13.6s/408 帧 → 5.0s/150 帧（-63%）。手段：去数字计数、精简配音文案、edge-tts `--rate` 加速。

**复现**：`cd mvp && node render.mjs scene.json`。

## 13. 多版式汇报片演进（2026-06-18）

在 Sprint 2 多页轮播基础上，从「单一 KPI 轮播」扩展为「结构化汇报片」：page 增加 `type`（`cover` / `cards` / `bars` / `table`），同一管线按类型生成内容区，模板 GSAP 通用化。

**验证**：12 页混合版式（2 cover + 5 cards + 1 bars + 3 table）汇报片端到端出片，各版式入场 / 显隐确定性正常，抽帧零残留。

**定位不变**：仍是信息型 / 数据型甜区（对手是 Excel / PPT），只是表达结构从"纯数字"扩展到"封面 + 卡片 + 进度条 + 表格"的完整汇报语言。详见 CHANGELOG [1.1.0]。
